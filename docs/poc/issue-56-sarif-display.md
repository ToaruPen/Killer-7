# PoC: SARIF Display Behavior for Category Splitting and Result Limits

- Issue: #56
- Related document: `docs/operations/sarif-reviewdog.md`
- Date: 2026-02-25
- Owner: @ToaruPen

## Goal

Measure how GitHub Code Scanning displays SARIF generated by Killer-7, focusing on:

1. Category splitting and filtering behavior
2. UI and API behavior around result-count limits

## Official Baseline

| Limit | Value | Official Behavior |
|---|---|---|
| SARIF file size | 10 MB | Rejected with `413 Payload Too Large` |
| Results per run | 25,000 | Rejected |
| UI display limit | 5,000 | Only the top 5,000 are shown |

References:
- https://docs.github.com/en/code-security/code-scanning/troubleshooting-sarif-uploads/file-too-large
- https://docs.github.com/en/code-security/code-scanning/troubleshooting-sarif-uploads/results-exceed-limit

## Method

Upload gzip + base64 encoded SARIF through the Code Scanning API:

`POST /repos/{owner}/{repo}/code-scanning/sarifs`

Upload script: `scripts/poc-sarif-upload.sh`

## Results

### 1. Category Splitting

Input: `category-split.sarif.json` with 20 findings split evenly across `P0`, `P1`, `P2`, and `P3`.

Observed:
- Distinct rules are displayed correctly
- Filtering by `ruleId` works
- Filtering by severity works
- Findings sharing a `ruleId` still appear as separate alerts through `partialFingerprints`

Observed analysis summary:

```text
Analysis: id=1004011776, results_count=20, rules_count=4, category=killer-7
```

### 2. Result Count Limits

#### `count-100`

- Upload: success
- Accepted: 100 findings

#### `count-1000`

- Upload: success
- Accepted: 1,000 findings

#### `count-5000`

- Upload: success
- Accepted: 5,000 findings

#### `count-5001`

- Upload: success
- API status: `processing_status: "complete"`
- Only 5,000 findings were retained for display
- No explicit error was returned

#### `count-10000`

- Upload: success
- Same silent truncation behavior as `count-5001`

#### `count-25000`

- Upload: rejected
- Practical failure was driven by the documented hard limits

#### `count-25001`

- Upload: rejected
- Same category of hard-limit failure

## Conclusions

- GitHub silently truncates display at 5,000 findings even when the upload succeeds
- The API does not make that truncation obvious enough for end users
- Killer-7 should warn before this point and fail fast above the hard limit
- Category splitting is safe and useful for PR-level review consumption
