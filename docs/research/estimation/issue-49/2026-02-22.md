## 3. 候補（必須: >= 5）

候補-1
概要: リリースタグ時に Docker image を publish し、管理PCは image digest 追従で更新
適用可否: Yes
仮説: 配布物の一意性（tag+digest）を基準に更新判定すれば no-op 判定とロールバックが容易
反証: 実行環境が Docker を前提にできない場合は運用負荷が増える
採否理由: 既存の Docker 実行導線（README/Dockerfile）と整合し、AC1-4を満たしやすい
根拠リンク:
- https://github.com/ToaruPen/Killer-7/blob/main/README.md
- https://github.com/ToaruPen/Killer-7/blob/main/Dockerfile
捨て条件:
- 管理PCでDocker運用が不可能
リスク/検証:
- リスク: publish失敗時に配布停止
- 検証方法: タグ作成→asset/image公開→更新ジョブ dry-run で検証

候補-2
概要: リリースタグ時に単体実行アセット（zip/tar）を配布し、管理PCで入れ替え
適用可否: Partial
仮説: Docker不要環境で導入しやすい
反証: 依存解決（Python/venv/パッケージ）を各PCで保証する必要がある
採否理由: AC1は満たせるが、AC4（自動ロールバック）の実装コストが上がる
根拠リンク:
- https://github.com/ToaruPen/Killer-7/blob/main/.github/workflows/release.yml
捨て条件:
- 管理PC間で実行環境差異が大きい
リスク/検証:
- リスク: 実行環境差異による起動失敗
- 検証方法: クリーン環境2種以上で smoke test

候補-3
概要: main 直追従の自動更新（定期pull）
適用可否: No
仮説: 常に最新機能を使える
反証: 未検証変更が全台へ即反映し、障害時の破壊半径が大きい
採否理由: 安全運用前提に反し、stable/canary 要件と相性が悪い
根拠リンク:
- https://github.com/ToaruPen/Killer-7/issues/49
捨て条件:
- 運用で即時安定性が求められる
リスク/検証:
- リスク: 全台同時障害
- 検証方法: canary先行運用との比較試験

候補-4
概要: stable/canary のチャネル設定を更新ジョブ側設定ファイルで管理
適用可否: Yes
仮説: チャネル分離により canary で問題検知して stable への波及を防げる
反証: 設定管理が曖昧だと誤配布が起きる
採否理由: AC3を直接満たす。設定ファイル1箇所に集約すれば運用も単純
根拠リンク:
- https://github.com/ToaruPen/Killer-7/issues/49
捨て条件:
- ホスト単位の設定管理ができない
リスク/検証:
- リスク: チャネル指定ミス
- 検証方法: stable/canary それぞれで意図したバージョンが選択されるE2E確認

候補-5
概要: 更新後 healthcheck 失敗時は直前バージョンへ即時ロールバック
適用可否: Yes
仮説: 更新処理を「pull→起動確認→昇格」の順にすれば安全に戻せる
反証: healthcheck が不十分だと障害を見逃す
採否理由: AC4の中核要件。ログ＋非0終了を合わせて fail-fast を担保できる
根拠リンク:
- https://github.com/ToaruPen/Killer-7/issues/49
捨て条件:
- ヘルスチェックコマンドが定義不能
リスク/検証:
- リスク: ロールバック漏れ
- 検証方法: 意図的に失敗する更新を注入し、復帰と終了コードを確認

## 4. 隣接領域探索（新規性が高い場合は必須）

隣接領域-1
概要: CIで生成物にハッシュ（SHA256）を残す配布モデル
根拠リンク:
- https://github.com/ToaruPen/Killer-7/blob/main/.github/workflows/release.yml

隣接領域-2
概要: 段階展開（canary→stable）と失敗時自動ロールバック
根拠リンク:
- https://aws.amazon.com/builders-library/ensuring-rollback-safety-during-deployments/

抽象化-1
原理/パターン: 配布チャネルを分けて破壊半径を制御

抽象化-2
原理/パターン: 更新処理は冪等（差分なしは no-op）

抽象化-3
原理/パターン: 更新後検証に失敗したら即ロールバックして停止

適用マッピング
- PRD: fail-fast と再現可能な運用ログの方針に一致
- Epic: 既存の安全性志向（検証重視）に整合
- Estimation: スクリプト実装＋workflow変更＋運用ドキュメント更新に分解できる
