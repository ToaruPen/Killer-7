## 3. 候補（必須: >= 5）

候補-1
概要: `sarif_export.py` に 25,000 件超過の fail-fast バリデーションを追加する。
適用可否: Yes
仮説: SARIF 変換直前で件数上限を明示エラー化すれば、GitHub 側 reject を待たずに原因を確定できる。
反証: 既存の findings 集約時点で件数が削減され、変換層でのチェックが実質無意味になる。
採否理由: PoC で 25,001 件が明示拒否されることを実測済みで、実装点も単一関数に局所化できる。
根拠リンク:
- https://docs.github.com/en/code-security/code-scanning/troubleshooting-sarif-uploads/results-exceed-limit
- https://github.com/ToaruPen/Killer-7/blob/main/docs/poc/issue-56-sarif-display.md
捨て条件:
- 集約層で常に 25,000 件未満に正規化される設計が別Issueで確定した場合。
リスク/検証:
- リスク: 例外文言が曖昧だと運用で再現手順を特定しづらい。
- 検証方法: `tests/test_sarif_export.py` に 25,001 件入力での `ValueError` 検証を追加。

候補-2
概要: 5,000 件超過時の「サイレント切り捨て警告」を `review-summary.md` / run metadata に反映する。
適用可否: Yes
仮説: Block ではなく警告として残すことで、運用者が「成功に見える部分反映」を識別できる。
反証: 警告出力先が不明瞭で、実行者が結果確認時に視認できない。
採否理由: PoC で 5,001 件以上は `processing_status: complete` のまま切り捨てられるため、Killer-7 側の明示警告が必須。
根拠リンク:
- https://docs.github.com/en/code-security/code-scanning/troubleshooting-sarif-uploads/results-exceed-limit
- https://github.com/ToaruPen/Killer-7/blob/main/docs/operations/sarif-reviewdog.md
捨て条件:
- 5,000 件超過を常に fail-fast にする方針へ変更された場合。
リスク/検証:
- リスク: 警告仕様を過剰に広げると PRD/Epic の範囲外になる。
- 検証方法: 5,001 件相当 fixture で warning の有無をユニットテストで固定。

候補-3
概要: 既存 inline 上限ガード（150 件）と同型の判定/メッセージ規約を SARIF 側に横展開する。
適用可否: Partial
仮説: `post_inline.py` の既存 fail-fast パターンを再利用すると、利用者向けメッセージの一貫性が上がる。
反証: inline と SARIF では「抑制」と「拒否」の意味が異なり、同型化し過ぎると誤解を招く。
採否理由: 実装パターン再利用は有効だが、終了コード/影響範囲は別扱いが必要。
根拠リンク:
- https://github.com/ToaruPen/Killer-7/blob/main/killer_7/github/post_inline.py
- https://github.com/ToaruPen/Killer-7/blob/main/docs/prd/killer-7.md
捨て条件:
- メッセージ統一よりも領域別の明示分離を優先する決定が出た場合。
リスク/検証:
- リスク: Block 条件を誤適用し、正常系を不必要に停止させる。
- 検証方法: `tests/test_post_inline.py` 既存挙動を回帰しつつ SARIF 専用テストを分離。

候補-4
概要: 運用ドキュメント（`docs/operations/sarif-reviewdog.md`）に PoC 結果の再現手順と判断表を追記する。
適用可否: Yes
仮説: 「成功/部分反映/失敗」の判定表を固定すれば、CI 実行者の解釈ブレを抑制できる。
反証: ドキュメント更新のみで CLI 出力が追随しない場合、実運用で参照されない。
採否理由: #57 の AC1/AC2 を満たすためにドキュメント集約は必須で、実装変更の前提にもなる。
根拠リンク:
- https://github.com/ToaruPen/Killer-7/blob/main/docs/poc/issue-56-sarif-display.md
- https://github.com/ToaruPen/Killer-7/blob/main/docs/operations/sarif-reviewdog.md
捨て条件:
- PoC 結果の前提（GitHub 側仕様）が再計測で覆った場合。
リスク/検証:
- リスク: 手順のコマンド例が現行 workflow と乖離する。
- 検証方法: `.github/workflows/poc-sarif-upload.yml` と整合チェック。

候補-5
概要: `.github/workflows/agentic-sdd-ci.yml` に SARIF 件数ガードレール検証（軽量チェック）を追加する。
適用可否: Partial
仮説: CI での早期検出により、運用手順だけに依存しない再現性を確保できる。
反証: 既存 CI 責務（test/lint/typecheck/docs）から逸脱し、実行時間増でノイズ化する。
採否理由: 効果はあるが #57 の最小スコープを超える可能性があるため、必要最小限に限定する前提で部分採用。
根拠リンク:
- https://github.com/ToaruPen/Killer-7/blob/main/.github/workflows/agentic-sdd-ci.yml
- https://docs.github.com/en/actions/using-workflows/workflow-syntax-for-github-actions
捨て条件:
- CI 追加が #57 の推定規模（50-150行）を超える場合は後続Issueへ分離。
リスク/検証:
- リスク: CI で実データ依存の不安定テストを入れてしまう。
- 検証方法: fixture ベースの決定的テストのみ採用し、外部 API 呼び出しを禁止。

候補-6
概要: `merge.py` 側で findings 件数メタ（総数・優先度別件数）を計算し、後段（SARIF/要約）へ渡す。
適用可否: Partial
仮説: 集約時点で件数メタを持てば、SARIF/inline 両方の警告・制限判定を統一できる。
反証: 既存 schema_version=3 の互換性に影響し、#57 の範囲を超える。
採否理由: 将来拡張として有用だが、今回の最小実装では変換層チェック優先が安全。
根拠リンク:
- https://github.com/ToaruPen/Killer-7/blob/main/killer_7/report/merge.py
- https://github.com/ToaruPen/Killer-7/blob/main/docs/epics/killer-7-epic.md
捨て条件:
- 既存 schema への項目追加が docs-sync で広範囲変更を要求する場合。
リスク/検証:
- リスク: 既存 consumer が未定義キーに依存し破綻する。
- 検証方法: `tests/test_cli.py` と summary schema 検証テストを回帰。

## 4. 隣接領域探索（新規性が高い場合は必須）

隣接領域-1
概要: GitHub Code Scanning の SARIF 件数上限を前提に、アップロード前に fail-fast / warning を分離する運用パターン。
根拠リンク:
- https://docs.github.com/en/code-security/code-scanning/troubleshooting-sarif-uploads/results-exceed-limit
- https://github.com/ToaruPen/Killer-7/blob/main/docs/poc/issue-56-sarif-display.md

隣接領域-2
概要: PR可視化の主経路（inline）と補助経路（SARIF/reviewdog）を分離し、品質ゲートを維持する段階導入パターン。
根拠リンク:
- https://github.com/ToaruPen/Killer-7/blob/main/docs/operations/sarif-reviewdog.md
- https://github.com/reviewdog/reviewdog

抽象化-1
原理/パターン: 外部プラットフォーム制約（表示上限/拒否条件）は、ツール側で明示的に検出して早期停止または警告に変換する。

抽象化-2
原理/パターン: 同じ「件数超過」でも致命条件（hard limit）と運用注意条件（display limit）を分けて扱う。

抽象化-3
原理/パターン: 実装変更と運用ドキュメント更新を同時に行い、再現手順と判定基準を固定する。

適用マッピング
- PRD: SARIF 出力要件（FR-6/FR-7）を維持しつつ、件数制約による誤解（サイレント切り捨て）を運用面で補完する。
- Epic: K7-19（SARIF/reviewdog 連携の段階導入）に対して、PoC 実測結果を安全側ルールとして反映する。
- Estimation: 変換層（sarif_export）・実行層（cli）・検証（tests）・運用文書（operations）に分割して見積もる。
