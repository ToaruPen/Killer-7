## 3. 候補（必須: >= 5）

候補-1
概要: `argparse.Namespace` を境界で明示型に変換する小ヘルパーを導入し、`Any` 伝播の起点を止める。
適用可否: Yes
仮説: `handle_review(args: argparse.Namespace)` の入口で `repo/pr/post/inline/...` を `TypedDict` 相当の構造へ吸収すれば、後段の `reportAny` がまとまって減る。
反証: 入口で型を固定しても `json.load` 系など別起点の `Any` が支配的で、warning 減少が限定的なら仮説は棄却。
採否理由: Issue AC1 が「argparse.Namespace 境界の型明示ヘルパー導入」を明示しており、SoTとの整合が最も高い。
根拠リンク:
- https://docs.python.org/3/library/argparse.html
- https://typing.python.org/en/latest/spec/typeddict.html
捨て条件:
- 境界ヘルパーの導入でコード可読性が大きく低下する場合。
リスク/検証:
- リスク: ヘルパーが過剰抽象化になり保守コストが増える。
- 検証方法: `basedpyright killer_7/cli.py` で `reportAny` の減少件数を比較。

候補-2
概要: `reportUnusedCallResult` は副作用専用呼び出しに `_ = ...` を明示して抑制する。
適用可否: Yes
仮説: `parser.add_argument(...)` や `dict.pop(...)` などの副作用目的呼び出しに `_ =` を付けるだけで AC2 を満たせる。
反証: `_ =` を付与しても可読性悪化が大きく、レビュールールで拒否されるなら棄却。
採否理由: basedpyright と pyright のメッセージが「意図的なら `_` へ代入」と明示しており、局所変更で安全。
根拠リンク:
- https://docs.basedpyright.com/v1.29.4/benefits-over-pyright/new-diagnostic-rules/
- https://github.com/microsoft/pyright/issues/8650
捨て条件:
- 代入が意味論を変える箇所（例えば式評価順依存）が見つかった場合。
リスク/検証:
- リスク: 付け忘れが残り warning が再発する。
- 検証方法: `basedpyright killer_7/cli.py tests/test_cli.py tests/test_sarif_export.py` を実行し `reportUnusedCallResult` の残件を確認。

候補-3
概要: `json.load` の戻り値を `object` として受け、`isinstance` で狭める fail-fast パターンへ統一する。
適用可否: Yes
仮説: `Any` を `object` + 明示ガードへ置換すると `reportAny` と `reportUnknown*` を同時に削減できる。
反証: ガード追加で分岐が過剰化し、推定変更行数が想定を超える場合は部分適用へ切替。
採否理由: リポジトリの fail-fast 方針（曖昧入力は明示エラー）と一致し、仕様外 fallback を増やさない。
根拠リンク:
- https://docs.python.org/3/library/json.html
- https://typing.python.org/en/latest/spec/special-types.html
捨て条件:
- 既存の JSON スキーマバリデーション層と重複し、冗長な検証になる場合。
リスク/検証:
- リスク: 型ガード漏れで未知の `Unknown` が残る。
- 検証方法: basedpyright warning diff と既存ユニットテストの green を確認。

候補-4
概要: テストコード（`tests/test_cli.py` / `tests/test_sarif_export.py`）は本体修正と分離し、型警告用の最小ユーティリティを追加する。
適用可否: Partial
仮説: テスト側の `json.loads` 結果アクセスを `Mapping[str, object]` キャストに寄せると warning を減らせる。
反証: キャスト増加でテスト可読性を損なう場合は、テスト側は段階導入（閾値緩和）で先送り。
採否理由: AC3 が「本体ロジック修正と切り離して実施」を要求するため、実装計画を分離する価値が高い。
根拠リンク:
- https://docs.python.org/3/library/unittest.html
- https://typing.python.org/en/latest/spec/special-types.html
捨て条件:
- テスト側修正が本体仕様変更を伴う場合（Issue スコープ逸脱）。
リスク/検証:
- リスク: 型整備でテスト意図が見えにくくなる。
- 検証方法: 対象テスト実行（`python3 -m unittest ...`）と warning 件数の分離レポート。

候補-5
概要: `pyproject.toml` に `[tool.basedpyright]` を導入し、段階導入の診断レベル（厳格/暫定）を明示する。
適用可否: Yes
仮説: ルールを設定ファイルへ固定すれば、ローカル環境差分を抑えつつ段階的に warning を減らせる。
反証: 設定追加後に既存 CI/ローカルでノイズが急増し、Issue範囲を超える場合は一部ルールを保留。
採否理由: AC4 の要求そのもの。現状 `pyproject.toml` に basedpyright セクションが存在しないため、可視化効果が高い。
根拠リンク:
- https://docs.basedpyright.com/latest/configuration/config-files
- https://github.com/microsoft/pyright/blob/main/docs/configuration.md
捨て条件:
- 既存運用で別の設定ファイル（`pyrightconfig.json`）が SoT の場合。
リスク/検証:
- リスク: ルール衝突で想定外 warning が増える。
- 検証方法: `basedpyright killer_7/cli.py tests/test_cli.py tests/test_sarif_export.py` の before/after 比較。

## 4. 隣接領域探索（新規性が高い場合は必須）

隣接領域-1
概要: 静的型ルールの段階導入（strictness ladder）を、CLI実装コードとテストコードで分けて運用する。
根拠リンク:
- https://docs.basedpyright.com/latest/configuration/config-files

隣接領域-2
概要: SARIF/reviewdog 運用ドキュメントと型チェック設定を連動させ、品質ゲートの説明責務をドキュメント化する。
根拠リンク:
- https://docs.github.com/en/code-security/code-scanning/integrating-with-code-scanning

抽象化-1
原理/パターン: 動的入力境界（CLI引数/JSONロード）で型を確定し、下流を純粋型で運ぶ。

抽象化-2
原理/パターン: warning 抑制は「ルール無効化」より「意図の明示（`_ = ...`）」を優先する。

抽象化-3
原理/パターン: 設定変更とコード変更を分離し、失敗時にロールバックしやすい最小単位を保つ。

適用マッピング
- PRD: Q6-5（機密保持）/Q6-7（性能）を維持しつつ、型チェック運用を明示化する。
- Epic: 単一CLI・シンプル優先方針のまま品質ゲートを段階強化する。
- Estimation: CLI本体・テスト・設定・運用ドキュメントの4タスクに分割して見積もる。
