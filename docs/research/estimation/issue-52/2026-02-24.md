## 3. 候補（必須: >= 5）

候補-1
概要: 既存Killer-7コアを維持し、`review-summary.json` から SARIF 2.1.0 へ変換して upload-sarif 連携する。
適用可否: Yes
仮説: 表示/連携層だけ標準化すれば、品質契約（verified/original_priority/Blocked条件）を維持したまま運用互換を上げられる。
反証: SARIFへの変換時に既存の優先度/evidence意味論が失われる、または既存テストが大きく破綻する。
採否理由: 既存の厳格スキーマと evidence ポリシーを残したまま追加実装できるため、リスク対効果が最も高い。
根拠リンク:
- https://docs.github.com/en/code-security/reference/code-scanning/sarif-support-for-code-scanning
- https://docs.github.com/en/code-security/how-tos/scan-code-for-vulnerabilities/integrate-with-existing-tools/uploading-a-sarif-file-to-github
- https://github.com/ToaruPen/Killer-7/issues/52
捨て条件:
- SARIF変換が 300LOC 超級の大規模改修になり、Issue粒度を満たせない場合は分割する。
リスク/検証:
- リスク: 位置情報のパス正規化差異で PR 上の表示が安定しない。
- 検証方法: 既存 fixture を使った変換ユニットテスト + サンプルPRで upload-sarif の表示確認。

候補-2
概要: reviewdog を `-f=sarif` または `-f=rdjson` でオプション導入し、PR注釈を補助経路として提供する。
適用可否: Yes
仮説: native inline 投稿をデフォルト維持しつつ reviewdog を追加すれば、外部エコシステムとの相互運用が向上する。
反証: reviewdog 経路が既存の fail-fast 条件（unmappable/over-limit）を回避してしまい、品質ゲートが曖昧になる。
採否理由: 補助経路としては有効だが、既存ポリシーの主経路を置き換えない前提でのみ採用可能。
根拠リンク:
- https://github.com/reviewdog/reviewdog
- https://github.com/reviewdog/reviewdog/blob/master/README.md
- https://github.com/reviewdog/reviewdog/blob/master/proto/rdf/README.md
捨て条件:
- reviewdog導入で既存inlineロジックを削る必要が出る場合は、段階導入方針に反するため見送る。
リスク/検証:
- リスク: コメント重複や reporter/filter-mode の設定不整合。
- 検証方法: ダミーPRで再実行し、重複件数と注釈位置の差分を比較。

候補-3
概要: GitHub Code Scanning を主UIとして採用し、PR注釈表示を Security結果中心に寄せる。
適用可否: Partial
仮説: SARIFアップロード結果を使えば、コード位置ハイライトの信頼性と履歴管理が向上する。
反証: PR表示条件（diff内の全行一致など）により、期待した運用と一致しないケースが多発する。
採否理由: 可視化基盤としては有効だが、Killer-7独自の Blocked 判定や投稿挙動を単独では置換できない。
根拠リンク:
- https://docs.github.com/en/code-security/code-scanning/managing-code-scanning-alerts/triaging-code-scanning-alerts-in-pull-requests
- https://docs.github.com/en/code-security/reference/code-scanning/sarif-support-for-code-scanning
- https://docs.github.com/en/code-security/code-scanning/troubleshooting-sarif-uploads
捨て条件:
- リポジトリ運用条件で code scanning の権限制約/制限により安定運用できない場合は補助採用に留める。
リスク/検証:
- リスク: SARIFサイズ/件数上限超過による取り込み失敗。
- 検証方法: 生成SARIFを段階的に増やし、上限時の失敗条件と縮退手順を確認。

候補-4
概要: Semgrep を追加して SARIF を生成し、Killer-7 findings と併走させる。
適用可否: Partial
仮説: Semgrep の検出結果を補助信号として使うことで、行位置の決定性を補強できる。
反証: ルール設計コストが見積もり範囲を超過し、Issue #52 の目的（導線整備）から逸脱する。
採否理由: 将来的な拡張候補として有効だが、Issue #52 の初期導入スコープには広すぎる。
根拠リンク:
- https://semgrep.dev/docs/cli-reference
- https://semgrep.dev/docs/customize-semgrep-ce
- https://github.com/semgrep/semgrep
捨て条件:
- 追加依存/運用負荷が増え、シンプル優先方針に反する場合は後続Issueへ分離。
リスク/検証:
- リスク: ノイズ増加で既存evidenceポリシーの説明責任が複雑化。
- 検証方法: PoCルールセットで false positive/true positive 比率を計測。

候補-5
概要: 既存実装のみで運用継続し、OSS連携は実施しない。
適用可否: No
仮説: 現行だけでも機能要件は満たすため、外部連携を増やさない方が安全。
反証: PR注釈/可視化の標準化ニーズが継続し、将来の運用コストが増加する。
採否理由: 現行品質は高いが、Issue #52 の目的（連携強化）を満たせないため不採用。
根拠リンク:
- https://github.com/ToaruPen/Killer-7/issues/52
- https://github.com/ToaruPen/Killer-7/blob/main/docs/prd/killer-7.md
- https://github.com/ToaruPen/Killer-7/blob/main/docs/epics/killer-7-epic.md
捨て条件:
- 連携要求が消滅し、ローカル成果物のみで十分と明確になった場合のみ再評価。
リスク/検証:
- リスク: 既存実装への技術的ロックイン。
- 検証方法: 四半期ごとに外部連携要件の再評価Issueを立てる。

候補-6
概要: SARIF変換ヘルパー（例: sarif-tools）を利用して、Killer-7固有データから中間変換を実施する。
適用可否: Partial
仮説: 変換専用OSSを使えばエッジケース対応を短縮できる。
反証: Killer-7固有フィールド（verified/original_priority等）の保持要件を満たせず、結局手書き変換が必要になる。
採否理由: ツールは参考になるが、最終的な意味論マッピングは独自実装が必要なため補助採用が妥当。
根拠リンク:
- https://github.com/microsoft/sarif-tools
- https://github.com/reviewdog/reviewdog/blob/master/proto/rdf/README.md
- https://docs.github.com/en/code-security/reference/code-scanning/sarif-support-for-code-scanning
捨て条件:
- 依存追加による保守負荷が、内製変換の工数削減効果を上回る場合は採用しない。
リスク/検証:
- リスク: 依存先バージョン差による変換互換性崩れ。
- 検証方法: lock付きでPoCし、同一入力で変換出力が安定するかを回帰テスト。

根拠リンク:

抽象化-1

原理/パターン: 主経路と補助経路を分離し、品質ゲート（Blocked条件）は主経路に一本化する。

適用マッピング
- PRD: FR-6/FR-7 の品質契約を維持しつつ、可視化経路を標準化できる。
- Epic: K7-10/K7-11/K7-14 周辺の実装意図を保ったまま、追加Issueで連携層を拡張できる。
